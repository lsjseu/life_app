## 1. 项目背景与目标

### 1.1 项目背景

随着人们健康意识的提升，越来越多的用户开始关注个人健康管理。然而，现有的健康管理App往往功能复杂、操作繁琐，难以满足用户随时随地、便捷记录的需求。"来福"小程序以微信小程序为起点，通过语音、图像等自然交互方式，为用户提供轻量化、智能化的健康管理服务。

### 1.2 产品愿景

成为用户最贴心的个人健康伙伴，通过AI大模型技术，帮助用户建立健康的生活方式。

### 1.3 核心目标

- **短期目标（3个月）**：完成微信小程序MVP版本上线，获取1000名种子用户
- **中期目标（6个月）**：完善核心功能，用户活跃度达到30%以上
- **长期目标（12个月）**：拓展至App和Web端，建立完整的健康管理生态

### 1.4 目标用户

| 用户画像 | 特征描述 | 核心需求 |
|---------|---------|---------|
| 职场白领 | 25-45岁，工作繁忙，饮食不规律，缺乏运动 | 便捷记录饮食运动，获取个性化健康建议 |
| 慢性病患者 | 有高血压、糖尿病等慢性病史，需要长期管理健康 | 记录身体状况，监控饮食运动，获取专业指导 |
| 健康关注者 | 注重养生，希望保持健康生活方式 | 系统管理健康数据，获取科学建议 |
| 中老年用户 | 对复杂操作不熟悉，需要简单易用的产品 | 通过语音、拍照等简单方式记录健康信息 |

## 2. 功能范围与边界

### 2.1 包含范围

- 用户健康档案管理（基本信息、病历、体检报告）
- 饮食记录与卡路里分析
- 运动记录与运动量评估
- AI健康建议与报告生成
- 语音、文字、拍照多模态交互

### 2.2 不包含范围（V1.0）

- 社交分享与社区功能
- 保险相关服务

## 3. 用户场景与用例

### 3.1 场景一：早餐记录

**用户**：职场白领小王

**场景**：早上上班路上，小王买了一份早餐，想记录下来

**操作流程**：
1. 小王打开"来福"小程序
2. 点击语音输入按钮，说："我吃了一个包子、一杯豆浆、一个鸡蛋"
3. 系统自动识别并记录，显示卡路里和营养成分
4. 系统给出早餐建议："早餐搭配不错，建议搭配一些蔬菜"

### 3.2 场景二：运动记录

**用户**：健身爱好者小李

**场景**：晚上跑步后，小李想记录今天的运动情况

**操作流程**：
1. 小李打开"来福"小程序
2. 选择运动记录，输入："跑步5公里，用时30分钟"
3. 系统计算消耗卡路里，评估运动量是否达标
4. 系统给出建议："今日运动量已达标，建议运动后补充水分和蛋白质"

### 3.3 场景三：体检报告分析

**用户**：慢性病患者张阿姨

**场景**：张阿姨拿到体检报告，想了解自己的健康状况

**操作流程**：
1. 张阿姨打开"来福"小程序
2. 点击上传体检报告，拍照上传
3. 系统自动识别报告内容，生成健康分析报告
4. 系统给出针对性建议："血糖偏高，建议控制碳水化合物摄入，每周监测血糖"

## 4. 功能详细描述

### 4.1 用户健康档案

#### 4.1.1 基本信息管理

**功能描述**：用户填写和管理个人基本健康信息

**界面元素**：
- 头像、昵称
- 性别、年龄、身高、体重
- 目标体重、活动水平
- 过敏史、家族病史

**业务规则**：
- 必填项：性别、年龄、身高、体重
- 身高体重用于计算BMI
- 活动水平分为：久坐、轻度、中度、重度

**权限控制**：仅用户本人可见和编辑

#### 4.1.2 病历管理

**功能描述**：用户记录和管理过往病历信息

**界面元素**：
- 病历列表（按时间倒序）
- 新增病历表单
  - 疾病名称
  - 诊断时间
  - 诊断医院
  - 病情描述
  - 治疗方案
  - 附件上传（诊断报告、处方等）

**业务规则**：
- 支持文字描述和图片上传
- 病历信息用于AI分析时参考

**权限控制**：仅用户本人可见和编辑

#### 4.1.3 体检报告管理

**功能描述**：用户上传和管理体检报告，系统自动分析

**界面元素**：
- 体检报告列表（按时间倒序）
- 上传入口（拍照/相册选择）
- 报告详情页
  - 原始报告图片
  - 结构化数据展示
  - AI分析报告
  - 健康建议

**业务规则**：
- 支持图片格式：JPG、PNG
- 系统使用OCR识别报告内容
- AI生成健康趋势分析（如有多次报告）

**权限控制**：仅用户本人可见

### 4.2 饮食记录

#### 4.2.1 多模态输入

**功能描述**：支持语音、文字、拍照三种输入方式

**界面元素**：
- 语音输入按钮（长按说话）
- 文字输入框
- 拍照按钮
- 输入预览区

**交互逻辑**：
- 语音输入：实时转文字，支持确认和编辑
- 拍照输入：拍摄食物图片，AI识别食物种类和分量
- 文字输入：支持自然语言描述

**业务规则**：
- 支持一次记录多餐
- 支持添加备注

#### 4.2.2 卡路里与营养分析

**功能描述**：自动计算摄入的卡路里和营养成分

**界面元素**：
- 今日总卡路里
- 三大营养素占比（碳水、蛋白质、脂肪）
- 每餐详情
- 与目标对比

**业务规则**：
- 基于食物数据库计算
- 考虑烹饪方式对热量的影响
- 目标基于用户基本信息计算

#### 4.2.3 饮食建议

**功能描述**：基于饮食记录和用户健康状况给出建议

**界面元素**：
- 今日饮食评分
- 优点总结
- 改进建议
- 推荐食物

**业务规则**：
- 结合用户病历、过敏史等信息
- 考虑季节、地域因素
- 建议具体可操作

### 4.3 运动记录

#### 4.3.1 运动录入

**功能描述**：记录用户运动情况

**界面元素**：
- 运动类型选择（跑步、游泳、瑜伽等）
- 运动时长输入
- 运动强度选择（低/中/高）
- 补充描述

**交互逻辑**：
- 支持语音输入："我今天跑步30分钟"
- 支持常见运动快速选择

#### 4.3.2 运动量评估

**功能描述**：评估用户当日运动量是否达标

**界面元素**：
- 消耗卡路里
- 运动时长
- 运动量评分
- 达标状态（未达标/达标/超标）

**业务规则**：
- 基于用户年龄、性别、体重计算
- 参考WHO推荐标准：成人每周150分钟中等强度运动
- 考虑用户日常活动水平

#### 4.3.3 运动建议

**功能描述**：给出个性化运动建议

**界面元素**：
- 今日运动评价
- 改进建议
- 推荐运动
- 注意事项（结合用户健康状况）

### 4.4 AI健康顾问

#### 4.4.1 健康对话

**功能描述**：用户可以与AI进行健康相关对话

**界面元素**：
- 对话界面
- 快捷问题入口
- 语音输入按钮

**交互逻辑**：
- 支持多轮对话
- 支持上下文理解
- 回答基于用户健康档案

#### 4.4.2 健康报告

**功能描述**：定期生成健康报告

**界面元素**：
- 周报/月报入口
- 报告概览
- 详细分析
- 行动建议

**业务规则**：
- 周报：每周一生成
- 月报：每月1日生成
- 包含趋势分析

### 4.5 个人中心

#### 4.5.1 数据概览

**功能描述**：展示用户健康数据概览

**界面元素**：
- BMI趋势图
- 体重变化曲线
- 运动达标率
- 饮食评分趋势

#### 4.5.2 设置

**功能描述**：用户设置

**界面元素**：
- 个人信息编辑
- 目标设置
- 通知设置
- 数据导出
- 账号安全

## 5. 非功能需求

### 5.1 性能需求

- 页面加载时间 < 2秒
- 语音识别响应时间 < 1秒
- AI分析报告生成时间 < 10秒
- 支持1000并发用户

### 5.2 安全性需求

- 用户数据端到端加密存储
- 医疗数据符合HIPAA/等保三级标准
- 支持数据脱敏展示
- 操作日志完整记录

### 5.3 兼容性需求

- 微信小程序：支持微信7.0+
- 未来App：iOS 12+、Android 8+
- 未来Web：支持Chrome、Safari、Firefox最新版本

### 5.4 可访问性需求

- 支持语音操作
- 字体大小可调节
- 高对比度模式
- 屏幕阅读器兼容

### 5.5 可扩展性需求

- 模块化架构，便于功能扩展
- 支持多端部署（小程序、App、Web）
- API设计考虑第三方集成

## 7. 技术架构建议

### 7.1 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                        客户端层                           │
├──────────────────┬──────────────────┬───────────────────┤
│  微信小程序      │   移动端App      │   Web端           │
└──────────────────┴──────────────────┴───────────────────┘
                            │
┌───────────────────────────▼──────────────────────────────┐
│                        API网关层                           │
└───────────────────────────┬──────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
┌───────▼───────┐  ┌───────▼───────┐  ┌──────▼────────┐
│  用户服务      │  │  健康数据服务  │  │  AI分析服务   │
└───────────────┘  └───────────────┘  └───────────────┘
        │                   │                   │
┌───────▼───────┐  ┌───────▼───────┐  ┌──────▼────────┐
│  关系型数据库  │  │  对象存储      │  │  大模型API    │
│  (MySQL)       │  │  (OSS)         │  │               │
└───────────────┘  └───────────────┘  └───────────────┘
```

### 7.2 核心技术选型

| 层级 | 技术选型 | 说明 |
|-----|---------|------|
| 前端（小程序） | 微信小程序原生开发 / UniApp | UniApp便于未来多端扩展 |
| 前端（App） | React Native / Flutter | 跨端开发 |
| 前端（Web） | React / Vue | 现代化前端框架 |
| 后端 | Node.js / Python / Go | 建议Python便于AI集成 |
| 数据库 | MySQL（主）+ Redis（缓存） | 关系型数据 + 高性能缓存 |
| 对象存储 | 阿里云OSS / 腾讯云COS | 存储图片、报告等文件 |
| AI服务 | OpenAI GPT / 文心一言 / 通义千问 | 大模型API |
| 语音识别 | 微信语音识别 / 阿里云ASR | 语音转文字 |
| OCR | 阿里云OCR / 腾讯云OCR | 识别体检报告 |

### 7.3 数据模型设计（概要）

#### 用户表 (users)
- id, openid, nickname, avatar, gender, age, height, weight, created_at, updated_at

#### 健康档案表 (health_profiles)
- id, user_id, allergies, family_history, created_at, updated_at

#### 病历表 (medical_records)
- id, user_id, disease_name, diagnosis_date, hospital, description, treatment, attachments, created_at

#### 饮食记录表 (diet_records)
- id, user_id, meal_type, food_items, calories, carbs, protein, fat, recorded_at, created_at

#### 运动记录表 (exercise_records)
- id, user_id, exercise_type, duration, intensity, calories_burned, recorded_at, created_at

#### 体检报告表 (medical_reports)
- id, user_id, report_date, report_image, structured_data, ai_analysis, created_at
